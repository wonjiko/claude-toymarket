# respond-review: PR 리뷰 대응 스킬

## 요약

GitHub PR에 달린 리뷰 코멘트를 수집하고, 각 코멘트에 대해 반영/미반영을 판단한 뒤, 유저 컨펌을 거쳐 코드 수정과 PR 코멘트를 자동으로 수행하는 스킬. 분석과 코드 수정 모두 서브에이전트 병렬 실행을 활용한다.

## 동기

PR 리뷰를 받은 뒤 각 코멘트에 대응하는 작업은 반복적이고 시간이 걸린다. 코멘트를 읽고, 코드를 확인하고, 반영 여부를 판단하고, 수정하고, 응답을 남기는 과정을 자동화한다.

## 왜 존재하는지

PR 리뷰 대응 워크플로우 전체를 하나의 스킬로 오케스트레이션한다. 기존 `code-review`(로컬 코드 리뷰)나 `make-pr`(PR 생성)과는 범위가 다르다.

## 언제 쓰면 안 되는지

- PR을 새로 만들 때 → `make-pr`
- 로컬 코드 품질 검증 → `code-review`
- 리뷰를 작성할 때 (이 스킬은 리뷰를 "받은 뒤 대응"하는 스킬)

## 설계

### 트리거

유저가 "리뷰 대응", "respond to review", "PR 리뷰 반영", "리뷰 코멘트 처리" 등을 요청할 때.

### 입력

- GitHub PR URL (또는 현재 브랜치의 PR을 `gh pr view`로 자동 탐지)

### 워크플로우

```
1. 초기 설정 (메인 세션, 순차)
   ├── PR URL 확인 (입력값 또는 현재 브랜치에서 자동 탐지)
   ├── 자동화 범위 질문: 커밋/PR 코멘트 자동 수행 여부
   └── gh api로 리뷰 코멘트 전체 수집

2. 분석 (서브에이전트, 병렬 — 코멘트 4개 이상 시)
   ├── 코멘트별 서브에이전트 디스패치
   │   └── 각 서브에이전트: diff 읽기 → 반영/미반영 판단 + 사유 + 근거 + 영향 범위 리포트
   └── 메인 세션이 결과 취합
   (코멘트 3개 이하: 메인 세션에서 순차 분석)

3. 유저 컨펌 (메인 세션)
   ├── 취합된 전체 리스트 + diff 함께 제시
   └── 유저가 수정할 항목만 고침 (나머지는 그대로 승인)

4. 그룹핑 (메인 세션)
   └── 2단계에서 보고된 영향 범위 기반으로 그룹핑
       충돌 가능성 있는 코멘트는 같은 그룹, 독립적인 것은 분리

5. 코드 수정 (서브에이전트, 병렬)
   └── 그룹별 서브에이전트 디스패치

6. 마무리 (메인 세션)
   ├── 수정 결과 취합/검증
   ├── 커밋 (자동화 선택 시)
   ├── PR에 코멘트 남김 (자동화 선택 시)
   └── lessons 축적
```

### 서브에이전트 프롬프트

#### 분석 서브에이전트 (2단계)

```
## 작업
PR 리뷰 코멘트를 분석하고 반영 여부를 판단하라.

## 코멘트 정보
- 작성자: {reviewer}
- 위치: {file_path}:{line_range}
- 내용: {comment_body}

## 해당 코드의 diff
{diff_hunk}

## 현재 파일 내용
Read 도구로 {file_path}를 읽어라.
추천 읽기 범위: {recommended_range} ({range_reason})
필요하다고 판단되면 더 넓게 읽어도 된다.

## 기존 판단 기준 (lessons)
{lessons_content}

## 판단 기준
- 버그 수정, 보안 이슈 → 반영
- 코드 컨벤션, 가독성 향상 → 반영
- 취향 차이, 트레이드오프가 있는 설계 의견 → 미반영 가능
- 현재 PR 범위 밖의 리팩토링 제안 → 미반영 가능
- 기존 lessons가 있으면 우선 참고

## 출력 (이 포맷을 정확히 따라라)
판단: 반영 | 미반영
사유: (1-2문장, 기술적 근거)
근거: (왜 그 판단이 적절한지, 코드 기반 설명)
영향 범위:
  - {file_path}:{function_name 또는 line_range}
  - (수정 시 영향받는 다른 파일이 있으면 추가)
수정 계획: (반영일 경우) 구체적으로 무엇을 어떻게 바꿀지
  (미반영일 경우) "없음"

## 제약
- 코드를 수정하지 마라. 분석만 하라.
- Read/Grep 도구만 사용하라.
- 추측하지 말고 코드를 읽고 판단하라.
```

#### 수정 서브에이전트 (5단계)

```
## 작업
유저가 승인한 PR 리뷰 코멘트를 코드에 반영하라.

## 반영할 코멘트 목록
{각 코멘트별:}
- 코멘트: {comment_body}
- 파일: {file_path}:{line_range}
- 승인된 수정 계획: {modification_plan}

## 제약
- 승인된 수정 계획에 명시된 범위만 수정하라.
- 수정 계획에 없는 추가 개선을 하지 마라.
- 수정 후 관련 파일의 import, 타입, 호출부가 깨지지 않는지 확인하라.
- Edit 도구를 사용하라.

## 완료 보고
수정한 파일 목록:
  - {file_path}: {무엇을 변경했는지 1줄 요약}
```

### 유저 컨펌 UI

```markdown
## PR 리뷰 대응 계획 (#123)

### 1. [반영] src/auth.ts:42 — @reviewer
> 원본 코멘트: "이 함수에서 null check가 빠져있습니다"

\`\`\`diff
- const user = getUser(id);
+ const user = getUser(id);  // 현재 코드
\`\`\`

**사유**: null 반환 가능성이 실제로 존재, 런타임 에러 위험
**근거**: getUser의 반환 타입이 User | null이고, 호출부에서 별도 처리 없이 user.name에 접근하고 있음
**수정 계획**: null guard 추가 후 early return

---

### 2. [미반영] src/utils.ts:88 — @reviewer
> 원본 코멘트: "이 헬퍼를 별도 모듈로 분리하면 어떨까요"

\`\`\`diff
  export function formatDate(d: Date) {  // 현재 코드
\`\`\`

**사유**: 현재 이 파일에서만 사용, PR 범위 밖 리팩토링
**근거**: Grep 결과 formatDate 호출이 utils.ts 내 1곳뿐, 분리 시 이점 없음
**수정 계획**: 없음

---

> 수정할 항목이 있으면 번호와 변경 내용을 알려주세요.
> 없으면 "확인"으로 진행합니다.
```

### PR 코멘트 포맷

각 리뷰 코멘트에 개별 reply로 남긴다.

- 반영 항목: `반영: {수정 내용 1-2문장 요약}`
- 미반영 항목: `미반영: {사유 1-2문장}`

### 판단 보정 시스템 (lessons)

#### 저장 구조

- `~/.claude/respond-review-lessons.md` — 유저 레벨, 워크플로우에서 자동 축적
- `plugins/skills-toybox/skills/respond-review/lessons.md` — 스킬 내장, 유저가 수동으로 옮긴 확정 규칙

#### 읽기 순서

1. 스킬 내장 `lessons.md` (기본 규칙)
2. `~/.claude/respond-review-lessons.md` (유저 개인 규칙, 오버라이드 가능)

두 파일의 내용을 합쳐서 분석 서브에이전트 프롬프트의 `{lessons_content}`에 포함.

#### 파일 포맷

```markdown
# respond-review lessons

## 반영 경향
- null safety 관련 지적은 항상 반영 (2026-04-21 학습)

## 미반영 경향
- PR 범위 밖 리팩토링 제안은 미반영 (2026-04-21 학습)
```

#### 축적 시점

워크플로우 마무리 단계에서, 유저가 판단을 수정한 항목이 있으면:
1. 수정 패턴 추출
2. 기존 lessons 파일에 비슷한 규칙이 있으면 업데이트, 없으면 추가 제안
3. 유저 컨펌 후 `~/.claude/respond-review-lessons.md`에 저장

#### 유저 안내

lessons 파일이 일정 크기 이상 축적되면 가끔 안내:
> `~/.claude/respond-review-lessons.md`에 규칙이 쌓이고 있습니다. 확정된 규칙은 스킬 내장으로 옮길 수 있어요.

### 자동화 범위 질문

워크플로우 시작 시 유저에게 확인:
> 코드 수정 후 자동으로 커밋하고 PR에 응답 코멘트를 남길까요? (Y/n)

- Y: 커밋 + PR 코멘트 자동
- n: 수정 결과만 보여주고, 커밋/코멘트는 별도 지시 대기
